Grundsätzliche Aufbau

Degraph nutzt JUnit-Test mit folgendem Aufbau

  • Definition des Classpath
  • includes und excludes
  • Definition von Slices
  • Regeln für die Abhängigkeit

Es prüft immer auf Zyklen!

Beispiel für Zyklenfreiheit


@Test
public void cycleFree() {
  assertThat(
      classpath().noJars().          // Keine Klassen aus Jars
      filterClasspath(TEST_CLASSES). // Filter für Tests
      printOnFailure(errorFilename). // Beim Fehler eine Grafik
      including(SZE_BASE_PACKAGE + "**"),
      is(violationFree()));          // Zyklenfreiheit
}

Technische Schichten


public void layer() {
  ConstraintBuilder testObject =
    customClasspath("./target/classes"). // Nur Produktivklassen
    printOnFailure(errorFilename).
    including(SZE_BASE_PACKAGE + "**").
    // Definition der Slices basierend auf ersten Package
    withSlicing("sze", SZE_BASE_PACKAGE + "(*).**").
    // Erlaube Abhängigkeiten
    allow(CONFIG_PACKAGE, SECURITY_PACKAGE,
       JLayer.oneOf(FRONTEND_LAYER, JOBS_PACKAGE),
       SERVICE_LAYER, DAO_LAYER, MODEL_PACKAGE,
       CONSTRAINTS_LAYER, UTIL_PACKAGE);
  assertThat(testObject, is(violationFree()));
}

Direkter Zugriff


@Test
public void layerDirect() {
  ConstraintBuilder testObject =
      customClasspath(PRODUCTION_CLASSES).
      printOnFailure(errorFilename).
      including(SZE_BASE_PACKAGE + FRONTEND_LAYER + ".**").
      including(SZE_BASE_PACKAGE + SERVICE_LAYER + ".**").
      including(SZE_BASE_PACKAGE + JOBS_PACKAGE + ".**").
      including(SZE_BASE_PACKAGE + DAO_LAYER + ".**").
      withSlicing("sze", SZE_BASE_PACKAGE + "(*).**").
      allowDirect(JLayer.oneOf(FRONTEND_LAYER, JOBS_PACKAGE),
          SERVICE_LAYER, DAO_LAYER);
  assertThat(testObject, is(violationFree()));
}

jQAssistant

In Java geschriebenes Tool, welches primär als Maven-Plugin genutzt wird.

    Der Ablauf ist dabei wie folgt:
  • importiere alle möglichen Struktur-Daten in Neo4j → scanner
  • reicher die Daten an → concepts
  • prüfe Regeln → constraints

Maven - Setup


<jqassistant.version>1.2.0</jqassistant.version>
…
<build>
  <plugins>
    <plugin>
      <groupId>com.buschmais.jqassistant</groupId>
      <artifactId>jqassistant-maven-plugin</artifactId>
      <version>${jqassistant.version}</version>
      <executions>
        <execution>
          <goals>
            <goal>scan</goal>
            <goal>analyze</goal>
          </goals>
          <configuration>
            <failOnViolations>true</failOnViolations>
          </configuration>
        </execution>
      </executions>
      <configuration>
        <reportProperties>
          <graphml.report.directory>
            ${project.build.directory}/graphml
          </graphml.report.directory>
        </reportProperties>
      </configuration>
      <dependencies>
        <dependency>
          <groupId>com.buschmais.jqassistant.plugin</groupId>
            <artifactId>jpa2</artifactId>
            <version>1.2</version>
        </dependency>
        <dependency>
          <groupId>com.buschmais.jqassistant.plugin</groupId>
          <artifactId>graphml</artifactId>
          <version>1.2</version>
        </dependency>
      </dependencies>
    </plugin>
  </plugins>
</build>

Konfigurationsmöglichkeiten

Konfigurationsdateien liegen im Order jqassistant und können folgende Formate haben:

xml oder ascii-doc

Zur Dokumentation kommen folgenden Regeln:

  • group - referenziert andere Regeln
  • concept - aggregiert Daten in der DB
  • constraints - definiert einzuhaltende Regeln

Die Regeln werden i.A. in Cypher geschrieben.

neo4j

neo4j ist eine Graph-Datenbank, d.h. es gibt Knoten und Kanten, die diese verbinden. Zum Beispiel:

  • (:Type) -[:DECLARES]-> (:Field)
  • (:Method) -[:READS]-> (:Field)

Cypher ist die Abfragesprache, analog SQL.

Cypher

Cypher-Abfragen entsprechen ASCII-Art, mit den Schlüsselwörtern

[MATCH … WHERE] RETURN [LIMIT].

Beispiel:


MATCH (m:Method) -[:READS]-> (f:Field)
RETURN m,f
LIMIT 15

Aufbau der Regeln


[[my-rules:MyConstraint]]❶
.A human readable description of the constraint.❷
[source,cypher,role=constraint,❸↩
requiresConcepts="my-rules:MyConcept",severity=blocker]❹
----
<Cypher-Query>❺
----

❶ - Die ID der Regel, des Blocks.

❷ - Title, des Blocks .Direkt gefolgt von einer Beschreibung.

❸ - Definiert einen Source-Block in der Sprach cypher, die role= weist es als constraint aus.

❹ - Benötigt ein concept und hat eine Severity.

❺ - Im Code-Block wird die Abfrage definiert.

concept: Setzen eines Labels


[[structure:RootPackage]]
.Labels the package `net.sf.sze` as `Root`.
[source,cypher,role=concept]
----
MATCH
  (root:Package)
WHERE
  root.fqn = "net.sf.sze"
SET
  root:Root
RETURN
  root as RootPackage
----

constraint: Struktur


[[structure:RootPackage]]
.All Types must be located within the root package↩
or a sub-package.
[source,cypher,role=constraint,requiresConcepts=↩
  "structure:RootPackage,maven:InternalFile"]
----
MATCH
  (type:Type:Internal)
WHERE NOT
  (:Package:Root)-[:CONTAINS*]->(type)
RETURN
  type as TypeOutsideRootPackage
----

Ablauf

1. Analyse starten mit 

mvn compile test-compile jqassistant:scan jqassistant:analyze

2. Server starten mit

mvn jqassistant:server

3. neo4j-Oberfläche starten

http://localhost:7474/browser/

4. cypher entwickeln und testen

Es gibt noch weitere Einsatzmöglichkeiten

  • Test-Jars bekommen den scope test
  • Spring-Nutzung ist korrekt
  • Testcoverage Regeln für bestimmte Methoden
  • Dein Bedarf

Referenzen auf Service-Implementationen


[[spring:ServiceImpl]]
.Check that classes in service.impl aren't referred by any class
[source,cypher,role=constraint,requiresConcepts=""]
----
MATCH (use:Type)-[:DEPENDS_ON]->(used:Type)
    <-[:CONTAINS*]- (serviceImpl:Package{fqn:"….service.impl"})
WHERE use <> used
AND NOT use.name CONTAINS "$"
AND NOT used.name CONTAINS "$"
// Die PDF-Konverter werden nicht von Spring verwaltet.
AND NOT used.name CONTAINS "OO2PdfConverter"
RETURN use.fqn AS FORBIDDEN_USER, used.fqn AS FORBIDDEN_USED
----

Bewertung - positiv

Folgende Punkte finde ich positiv

  • Läuft gut im Continously-Build
  • umfangreiche Dokumentation
  • extrem Leistungsstark
  • erweiterbar
  • man erwirbt Know-How über neo4j
  • Kostenlos, daher auch beim Kunden eher möglich
  • Sprach unabhängig (siehe git oder maven)
  • Debugbar

Bewertung - negativ

Folgende Punkte sehe ich kritisch

  • Laufzeit im Minutenbereich
  • Einarbeitungsaufwand
  • kleine Community

Fazit

Strukturanalyse lohnt sich, da die Software wartbarer bleibt. Zumindest Degraph sollte daher eingesetzt werden.

Mit jQAssistant sind umfangreiche Analysen jenseits der Abhängigkeitsanalyse möglich. Der Einsatz erfordert allerdings eine gewisse Einarbeitung.
